# PixelForge 工作区约定（所有 agent 必读）

## 0. 对话任务角色与授权边界

PixelForge 日常只保留一个长期规划任务 `P` 和一个当前执行任务 `E`；发布任务 `L` 仅在需要时临时创建。用户可以把问题、截图和体验反馈统一发给 `P`，无需自行整理工单或复制到 `E`。

### 0.1 P：总控、规划与反馈入口

- `P` 负责恢复项目事实、诊断问题、判断产品影响、维护路线与决策、汇总状态，并把明确的执行事项转交给 `E`。
- `P` 默认只读：可以阅读代码/文档、运行非破坏性诊断、自动化和日志检查，也可以把临时证据写入已忽略的 `scratch/`；不得修改 `pixel/`、不得构建候选、不得 stage/commit，也不得指派子 agent 写产品代码。
- 只有项目所有者明确批准某项规划/规则文档变更时，`P` 才可修改 `pixelforge-plan/` 或既有根规则文件；该批准不自动授权产品代码实现。
- 用户在 `P` 中提问、质疑、提供截图或说“为什么会这样”，默认都是诊断与反馈，不是 `P` 自行实施修复的授权。
- 用户说“只诊断”或“先别修”时，`P` 不得转交实现；用户说“暂停执行”时，停止向 `E` 增加新事项并保留现场。

### 0.2 E：当前唯一代码执行者

- `E` 只执行项目所有者已批准的执行任务或修复窗口，负责代码、测试、允许范围内的 i18n/报告、分支/worktree、集成和工程结论。
- 一个修复窗口可以一次性授权一组后续反馈，例如“本轮人工验收发现的、未改变冻结产品含义和契约的 Beta 0.6 缺陷”。窗口开启后，`P` 可将属于该范围的明确缺陷直接转交 `E`，无需逐 bug 再请求批准。
- `E` 只能在已批准范围内修复实现问题和清零相关自动化红灯；不得自行改变冻结设计、跨模块契约含义、版本范围或用户旅程。遇到这些情况必须把证据与选项退回 `P`/项目所有者决定。
- 同一执行事项只有一个 `E` 主任务和集成人。并行写代码的子 agent 必须使用独立 worktree/分支和不冲突的写入范围，最终由 `E` 审核合流。
- `E` 的工程通过只返回 `P` 汇总；项目所有者人工签收、merge、push 和发布仍是独立权限。

### 0.3 P 与 E 的快速反馈回路

```text
项目所有者向 P 反馈
  -> P 诊断、去重并判断范围
  -> 明确且属于已开启修复窗口：直接转交 E
  -> 涉及产品取舍、契约变化或范围扩张：只向项目所有者请求一次决定
  -> E 实现并自动验证
  -> P 汇总结果，等待项目所有者复验
```

- `P` 和 `E` 不得在同一任务内互相升权；从诊断进入实现必须由独立 `E` 承接。
- 只有三类动作必须另行获得项目所有者明确批准：改变产品/体验或冻结规格、修改跨模块契约或扩大范围、merge/push/发布。数据丢失、密钥泄漏和许可证风险必须立即暂停并报告。
- 独立审查优先由 `E` 使用只读子 agent 完成；只有高风险或需要独立人工判定时才创建临时审查任务。审查者只报告问题，不顺手修复。

### 0.4 上下文压缩、交接与异常恢复

- 聊天记忆不是事实来源。冻结决定必须落在计划、规格或 `02-contracts/`；活动状态由 Git 实态和已忽略的 `scratch/thread-handoffs/` 极简交接共同恢复。
- 每个活动 `P/E` 交接至少记录：角色、修复窗口/任务 ID 与授权状态、基线 SHA、分支/worktree、既有 dirty 文件、允许/禁止范围、冻结决定来源、当前问题、完成提交、验证结果、已知失败和下一步。
- 首次进入或上下文压缩后，先重读本文件、`README.md`、`CURRENT-STATE.md`、当前计划/冻结规格、相关契约与质量文档，再核对 branch/HEAD/worktree/dirty 和交接记录；恢复结果不一致时先停下报告。
- 若无法证明当前任务拥有代码写入授权，自动降级为 `P` 的只读诊断权限；上下文压缩永远不能扩大权限。
- 若 `P`、诊断者或审查者误改产品代码，必须立即停止，不继续“顺便做完”、不 stage/commit、不清理现场；记录 status/diff，标记为未授权候选，等待项目所有者决定保留、迁移或放弃。

### 0.5 技术说明与进度反馈

- 无论是 `P` 规划、诊断，还是 `E` 实现、测试，都必须及时用十分通俗易懂的语言说明当前进度、发现的问题、对用户的影响、下一步和是否需要决定；不能只汇报文件名、术语、测试数字或代码细节。
- 技术名词首次出现时要顺带解释“它是什么、为什么与当前问题有关”。反馈应让不了解 PixelForge 历史、代码和工具链的人也能接手，至少能看懂：原问题是什么、为什么发生、已经决定什么、改了什么/没有改什么、证据与风险、接下来做什么。
- 规划涉及技术细节或存在真实可选方案时，必须列出可行选项，说明推荐哪一个、推荐理由、各自优点与代价，以及对用户体验、开发量、兼容性、测试和后续维护的影响；不得把未经批准的技术取舍写成既定事实。
- 如果并不存在需要项目所有者选择的真实权衡，应直接说明原因并给出明确建议，不为了显得全面而制造伪选项。
- 大段源码、原始日志和完整 diff 只作为可查证证据，不代替通俗结论；先给人能读懂的摘要，再给文件、命令和数字。

## 1. 文件范围与安全红线

- 代码只写入 `pixel/`；计划、研究与报告只写入 `pixelforge-plan/`；临时草稿写入已忽略的 `scratch/`。除既有工作区规则文件外，不在仓库根目录新增文件。
- 里程碑/修复报告写入 `pixelforge-plan/03-milestones/reports/`；算法研究写入 `04-research/`；外部算法参考写入 `06-algorithm-refs/`。
- 不修改、不纳入 git：`垃圾桶/`、`godot-interactive-guide/`。
- `test picture/` 与 `pixel/tests/fixtures/real/` 含未获公开许可的图像，绝不 commit 或 push，也不得绕过 ignore 规则。

## 2. 产品与工程默认规则

- 新功能先说明它服务哪段用户旅程；“完成”表示入口、过程反馈、结果和下一步在应用中连通，不能只以代码或单测存在为准。
- `pixelforge-plan/02-contracts/` 是跨模块接口的唯一事实来源。发现契约缺陷时提出修订并等待批准，禁止静默绕过。
- 用户可见文案必须经过集中 i18n 目录/访问层；不得在组件中散落裸字符串。`pixel/ui/shell/strings.gd` 可作为迁移期兼容入口。
- UI chrome 继承根窗口统一缩放策略；画布美术继续按设备像素整数对齐自管。不要在组件内私设第二套缩放倍率或硬编码像素字号；具体规则与守护见 `ARCHITECTURE.md §5` 和 `pixel/scripts/check_ui_scaling.sh`。
- 禁止裸 `print`；使用项目日志工具，并避免输出密钥、用户素材或完整外部响应。
- 注释只保留在关键逻辑节点：模块职责、契约边界、非显然不变量、兼容/安全原因，以及影响应用行为的重要实现决策。不要逐行复述代码、记录临时思考或用长注释代替文档。
- 文件约 1000 行是软目标；只按职责拆分，不为压行数破坏内聚性。
- 新增外部算法或资产参考时记录来源与许可证。既有 perfectPixel 集成说明见 `06-algorithm-refs/perfectPixel/INTEGRATION.md`。

## 3. 开发节奏与验收

- 任务卡仍是边界清晰的实现单元；仅在项目所有者已经明确批准的 Beta 0.6 `E` 执行任务/修复窗口中，一个 Goal 才可连续完成 B6-0 至 B6-8、无需在单卡或集成切片停下等待用户。该规则不得被 `P`、诊断或审查任务解释为代码写入授权。
- 开发中持续运行定向自动化测试；在集成边界运行全量回归。自动化红灯不得带入下一阶段。
- Beta 0.6 开发期间完全不使用 Computer Use，也不请求项目所有者分段测试。只允许自动化测试和脚本生成的固定截图；B6-0 至 B6-8 全部工程完成后再由项目所有者统一人工验收。
- 报告必须区分：`工程通过`、`人工通过`、`发布通过`。自动化与脚本截图不能替代项目所有者签收。
- UI、缩放、字体、跨平台等难定位问题先通过自动化矩阵、日志和脚本截图建立证据与根因，再由已获授权的 `E` 在隔离分支实施最小修复；`P` 只负责诊断与转交。禁止把未经验证的猜测写成既定方案。
- 用户否决的原型必须标记为未通过、撤销或登记设计债，后续不得描述为已完成能力。
- 产品方向、体验取舍或契约含义存在实质歧义时，把选项、依据和影响交给用户决定。
- 子 agent 适合做原始资料收集、历史/源码定位和边界明确的并行实现；主 agent 负责产品判断、技术结论、集成与最终验收口径。
- Beta 0.6 的 `1A / 2B / 3C` 已由 `BETA-0.6-CARD-DESIGN-SPEC.md` 冻结，不得重新调研后换方案。只有规格与现有实现出现无法判断的真实冲突时，才允许低上下文子 agent 只读核对 `hero8152/Infinite-Canvas` 单一对应行为；不得复制其代码、样式、截图、资产或工作流文件。
- 交接只记录当前阶段、关键提交、验证命令、已知失败和下一步，避免重复粘贴大段源码或历史报告。

## 4. Git 纪律

- 开发在本地 `codex/` 分支或独立 worktree 进行；按“卡片基础 / 工作区与输入 / 生成与结果 / LOD 与最终证据”整体切片保留可回退提交。Beta 0.6 只在 B6-8 生成一次 diff 模式工程报告与唯一候选。
- 项目所有者统一人工验收通过前，不合并到 `main`、不 push。若候选不通过，由同一获批 `E` 修复窗口优先修复当前未推送分支；`P` 不得据此自行接管实现。需要整体放弃时保留证据并等待用户确认后再执行破坏性操作。
- 多 agent 并行改代码时各用独立 worktree 与分支，完成后由主 agent 审核并集成。
- commit 前检查 staged 文件；下列命令应无输出（`addons/gut` 自带图标除外）：

  ```bash
  git diff --cached --name-only | grep -iE '\.(png|jpg|jpeg)$'
  ```
