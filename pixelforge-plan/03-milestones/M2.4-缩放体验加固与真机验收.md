## 0. 如何使用本卡

- 本卡是 M2.3 的"收口卡"：M2.3 把地基浇了，本卡负责**真机验真 + 补三处实锤缺陷 + 把多屏纳入**。每个 Fix 含：现状（实锤）→ 根因/证据 → 修复方向 → 验收。
- 行号会漂，**一律以符号名为准**（如 `infinite_canvas.gd::_update_layer_transform`）。
- 涉及"决策翻转/规范改动"的（尤其 Fix D 触及 M2.3 已批准的方案2），走**"提议待批准"**，标注清楚，Ruo 批了再落。
- 沿用既有约定：诚实工程（延后项显式登记）、完成报告 git diff 模式、图片红线（`test picture/`、`tests/fixtures/real/` 未授权 AI 图绝不 commit/push）、禁裸 `print`（用 `log_util`）、UI 串集中 `ui/shell/strings.gd`、`gdformat`/`gdlint` 零告警。
- **VM 限制**：本环境无 Godot 二进制，渲染/原生部件/跨平台只能真机走查。凡标【真机】的步骤，施工 AI 在 VM 内只能改代码 + 备齐自测脚本，**最终签收必须 Ruo 在真机执行并回填矩阵**。

## 1. 护栏（违反即返工）

- **不回退 M2.3 的正确部分**：`content_scale_factor` 骨架、`canvas_scale_policy.gd` 纯函数、Wiring A 的坐标系统一、`check_ui_scaling.sh` 守护——都保留，本卡是**叠加加固**，不是推倒。
- **core 算法 / 图模型 / 契约一律不动**。本卡纯表现层 + 启动/窗口策略；`canvas.json`/GRAPH-SCHEMA/PROJECT-FORMAT 不变，不升版、不写迁移。
- **检测启发式继续复用、不重写**：`compute_auto_interface_scale` / `should_use_macos_retina_fallback` / `fit_interface_scale_to_startup_screen` 逻辑不动；本卡只改"算出来的 F 如何被消费"（Fix D 可能在其输出后加一道整数吸附，属新增、非改它们本体）。
- **Fix D / Fix E 触及已批准决策与新增运行时行为，先提议后落**：Fix D 的"chrome 是否就近整数吸附"实质是对 M2.3「方案2（分数 chrome）」的修订；Fix E 的 live re-scale 是 M2.3 §8 显式延后项的撤销。两者都标 **[待 Ruo 批准]**，真机 spike 给出实测严重度后再定最终方案。
- **别让旧雷复发**：内屏黑窗钳制（M2.2 P0-1）、清洗台滚动可达（M2.2 P0-2）、三向缩放同步、导入未激活工具仍可拖画布——均有回归测试，改完跑 smoke 确认。

## 2. 背景：为什么需要 M2.4

M2.3 复核（2026-06-16）结论：**实现质量高，但"修复"未完成，因为成立性押在被跳过的那一步上**。具体：

- **门禁被跳过**：M2.3 Phase 0 写明是真机 mac Retina + Win150% 的 go/no-go spike，还配了回退条款（FileDialog 不随 factor→暂停迁移）。实际 Phase 1–4 做完直接提交 `ab54b0c`，Phase 0 与 Phase 5 真机矩阵一起推后。**当前 main 上的缩放从未在真机上被验证过。**
- **headless 结构性盲区**：`--headless` 禁渲染，跑得了 `canvas_scale_policy` 的数学和 `content_scale_factor==1.5` 的属性赋值，**测不出**：chrome 是否真被放大、画布平移后有无接缝、原生 PopupMenu/FileDialog 行为、非整数档字体是否糊。M2.3 的 108 个测全绿只证明了**坐标代数对**，不证明**像素渲染对**。
- **三处实锤缺陷**（详见 §3 Fix B/C/D）：画布缺像素平移对齐（"无接缝"是过度声明）、原生弹窗在 4.6.3 可能有功能性 bug、Windows 非整数 factor chrome 有糊风险。
- **一处被低估的延后**：M2.3 把多屏运行时重算列为延后，但该迁移恰好从 `content_scale_mode=DISABLED`（多 DPI stretch 错位的唯一已知规避）主动搬进了 factor 启用区——延后多屏 = 踏进已知 bug 区还没兜底。Ruo 已决定本卡纳入（Fix E）。

一句话目标：**让 PixelForge 在 mac Retina 与 Windows 上，缩放这件最基础的事，经真机签收、确实正确，再开 M3。**

## 3. 缺陷与修复方案

### Fix A 【真机】补做被跳过的 go/no-go spike（本卡前置，定其余 Fix 的实际严重度）

- **现状**：M2.3 Phase 0 未执行，真机表现未知。
- **做法**：在真 mac Retina(2x) + Win 125%/150% 上，跑当前 `ab54b0c` build + 一个最小验证场景（一个 Panel + 一个 FileDialog + 一个 PopupMenu + 一张画 16×16 已知棋盘的画布）。三档 F（1.0/1.5/2.0）分别量：
  1. **chrome 真放大**：DISABLED + `content_scale_factor=1.5/2.0` 下，面板/按钮/字号是否按 F 等比放大且清晰（验证 DISABLED+factor 这个惯用法在 4.6.3 真的渲染缩放，而非只设了属性）。
  2. **画布接缝**：16×16 在 100% zoom、且**平移到非整数 camera_center 后**，每美术像素是否恰好 `F_canvas` 物理像素、有无 N±1 宽度行（直接暴露 Fix B）。
  3. **原生部件**：FileDialog / PopupMenu 是否随 factor 一致放大、命中区是否对齐（直接暴露 Fix C）。
  4. **跨屏启动**：把窗口分别在 Retina 屏与外接屏启动，检测的 F 是否各自正确（为 Fix E 打基线）。
- **验收**：产出三档实测截图 + 缺陷清单 + 每项 go/no-go，回填本卡 §5.4 矩阵 mac/Win 列。**若发现 chrome 在 DISABLED+factor 下根本不缩放**→ 触发应急（切 `content_scale_mode=CANVAS_ITEMS` 重试，见 Fix D 备选），并立即回报 Ruo。

### Fix B 【代码+真机】画布像素平移对齐，消接缝（P1）

- **现状（实锤）**：`infinite_canvas.gd::_update_layer_transform` 里 `item_layer.scale = Vector2.ONE * _get_art_logical_scale()`（净倍率整数，OK），但 `item_layer.position = size*0.5 - camera_center*scale`。`camera_center` 平移后通常为小数 → item_layer 落在亚像素位置 → NEAREST 下相邻像素块出现 N 与 N±1 宽度不等的行。项目里 grep 不到任何 `snap_2d_*`/`gpu_pixel_snap`，`project.godot` 也未设。
- **根因**：M2.3 §3.3 不变量只约束了**放大倍率**为整数，未约束**平移**为整数物理像素。整数倍率是无接缝的必要非充分条件。证据：研究指像素美术整数缩放还需 transform 吸附；Godot 社区像素画标准做法含 `snap_2d_transforms_to_pixel`。
- **修复方向（二选一，推荐局部）**：
  - 局部（推荐）：在 `_update_layer_transform` 里，把 `item_layer.position` 对**物理像素**取整后再换算回 Control 空间——即让 `camera_center*scale` 的结果落在整数物理像素网格（考虑 F：物理 = 逻辑×F，取整后除回）。只作用画布层，不影响 UI 动画/相机。
  - 全局（备选）：`project.godot` 设 `rendering/2d/snap/snap_2d_transforms_to_pixel=true`（+ `snap_2d_vertices_to_pixel`）。简单但**影响所有 2D**，可能让 UI 平滑动画/overlay 抖动，须真机核 chrome 没被吸出问题。
  - `camera_zoom<1`（缩小看全图，下采样）不强制整数对齐。
- **验收**：① 纯函数测——给定 camera_center/scale，断言对齐后的 position 落在 `1/F` 物理像素网格（headless 可测数学部分）；② 【真机】已知 sprite 100% zoom 平移任意小数距离，截图每像素块均匀 = `F_canvas`、无 N±1 行；③ 同步修正 M2.3 §3.3 文案，把"整数→无接缝"改为"整数倍率 + 平移像素对齐→无接缝"。

### Fix C 【真机+代码】原生 PopupMenu / FileDialog 策略（P1）

- **现状/风险（研究锤）**：`content_scale_mode/factor≠1` + 非嵌入子窗口下，Godot 有一串已证实 issue：PopupMenu 下半区点不动（#102159）、MenuButton hover 命中区偏移（#86240）、Popup 不继承父级 scale（#82335）。修复 PR **#104399**（Popup `content_scale_factor` 来源由 parent Window 改 parent Viewport）2025-03-28 才并入 master。
- **做法**：
  1. **先查版本**：确认项目 Godot **4.6.3** 是否含 #104399 的 cherry-pick（看 4.6 分支提交 / release notes）。结论写进卡。
  2. **PopupMenu 策略**：若 4.6.3 **不含** #104399 → Retina 上菜单是功能性 bug。两条出路：(i) 升 Godot 到含修复的 4.6.x；(ii) 把弹出菜单改**嵌入式子窗口**（embed subwindows，项目级或 Viewport 级开关），使其走 viewport 变换、规避原生路径 bug。真机确认后二选一。
  3. **FileDialog 策略**：macOS 原生对话框**不吃** `content_scale_factor`（OS 绘制，#82531 另记原生显示坑）——这是预期不是 bug。决定：接受原生外观（与系统一致），或 `use_native_dialog=false` 用 Godot 自绘以受缩放控制 + 风格统一。真机看了当前观感再定。
- **验收**：【真机】mac Retina + Win 下打开**所有**菜单/文件对话框，命中区对齐、可点、缩放一致；卡内记录 #104399 结论 + 选定策略。

### Fix D 【真机定性 + 代码】Windows 非整数档 chrome 不糊（P2）　**[待 Ruo 批准]**

- **现状/风险**：整数对齐只做在画布；chrome 直接吃原始 F。Win 125%/150% → `content_scale_factor=1.25/1.5`，Godot 非整数 factor 下字体/Tooltip 有糊风险（#57340、#96495）。**但**：当前 `content_scale_mode=DISABLED` 下 factor 兼作 font oversampling，1.5 字体**未必**糊——须真机定性，别先入为主。
- **决策分叉（改 M2.3 方案2 需批准）**：
  - **先定性**（Fix A 顺带量）：真机看 Win150% chrome 是否真糊。**不糊就不改**，只在卡里登记"已验、可接受"。
  - 若糊，三选一：
    - **(i) chrome 就近整数吸附**【推荐】：把消费端 `content_scale_factor` 取 `max(1, round(F))`（Win150%→2.0、125%→1.0）。**顺带巨大简化**：F 恒整数 ⇒ `F_canvas==F` ⇒ 画布补偿 `F_canvas/F==1` ⇒ Wiring A 退化为 `item_layer.scale=camera_zoom`，接缝风险同步消失。代价：125% 被吸到 1.0（界面偏小）或 150% 吸到 2.0（偏大），缩放档变粗。**这是对 M2.3「分数 chrome」方案2 的修订，必须 Ruo 批。**
    - **(ii) 切 `content_scale_mode=CANVAS_ITEMS`**：分数 factor 下 CANVAS_ITEMS 在分辨率层拉伸，字体比 DISABLED/VIEWPORT 更清晰。保留分数 chrome，但需重验所有 Control 布局（mode 改变会动锚点/参考尺寸语义），回归面较大。
    - **(iii) 手动整数档兜底**：保留检测分数值，但"界面缩放"设置允许用户在 100/150/200 手选，默认就近整数。
  - **推荐组合**：(i)+(iii)——整个 app 用一个整数 F（简化 + 不糊 + 不接缝），设置里留手动档兜底极端 DPI。
- **验收**：【真机】Win 100/125/150 三档 chrome 字体边界锐利、无糊；若选 (i) 则补测画布在整数 F 下不变量仍成立、并更新 §8 规范草案（"界面 factor 就近整数"）。

### Fix E 【真机重 + 代码】多屏运行时重标定 / live re-scale（P2，本卡新纳入）

- **现状**：M2.3 只保证启动按当前屏检测；跨屏拖动不重算。研究 #111271：跨不同 DPI 屏 stretch 错位，禁用 stretch 是唯一已知规避——而本项目已搬出 DISABLED-only 状态，故必须主动兜底。
- **做法**：
  1. **监听屏幕/DPI 变化**：监测当前屏变更（如轮询 `DisplayServer.window_get_current_screen` + `DisplayServer.screen_get_scale` 的变化，或接 macOS 的 DPI 变化通知——**确切信号名须在真机/4.6.3 上确认，研究指 `dpi_changed` 为 macOS-only 且需手处理**）。
  2. **重标定**：变化时重算并重设 `content_scale_factor`（经 Fix D 的整数策略）、`F_canvas`、`item_layer` 补偿；chrome 用逻辑常量 + factor 自动重排，但**派生缓存**（主题字号 override、缓存的 min_size 等）需重建。
  3. **稳健性**：**防抖**（连续 resize 事件合并）+ **防重入/防崩**——研究 #98514：Windows 运行时 DPI 变更曾致 Godot 4.3 100% CPU/崩溃。重标定路径要可关（临时设置 `ui.live_rescale=on|off`，默认 on，真机出问题可秒退回启动期重标定）。
- **验收**：【真机】mac 双屏（Retina↔1x）、Win 主屏↔外接（不同缩放），拖窗跨屏后界面与画布**即时**正确重标定、不崩、不抖、锚点不乱；无副显示器时零回归。

## 5. 调试与测试参考

### 5.1 Headless 能守住的（CI/agent 可跑）

- Fix B：position 对齐纯函数测（给 camera_center/scale，断言落 `1/F` 物理像素网格）。
- Fix D（若选整数吸附）：`snap_interface_scale(F)->int` 纯函数测（1.0→1，1.25→1，1.49→1，1.5→2，2.0→2），并补"整数 F 下 `compute_canvas_compensation_scale==1.0`"。
- 保留 M2.3 全部 108 测不回归（尤其 zoom 锚点、pan、device_scale）。
- **明确边界**：以上只证数学/属性，**不证渲染**。Fix A/B（接缝）、C（命中区）、D（糊）、E（重标定）的成立性**只能真机签收**。

### 5.2 静态守护扩展（check_ui_scaling.sh）

- 保留现有四条（禁 `_scaled_int`/`_scaled_vec2`、禁未豁免 `ui_scale`、禁 `custom_minimum_size` 数字字面量、禁字号数字字面量）。
- 新增（按所选方案）：若 Fix B 走局部对齐，加一条"`item_layer.position =` 行必须经对齐函数"的弱守护（grep 提醒）；若 Fix D 选整数吸附，加"`content_scale_factor =` 赋值处必须经 `snap_interface_scale`"。
- 合法例外仍用行尾 `# scale-exempt: <理由>`。
- **守护性质提醒**：这是**负向 lint**（禁旧模式），过了不等于缩放对——真机矩阵才是正向证据。别让绿色守护制造虚假信心。

### 5.3 真机调试支撑

- 扩 `Log.info("Interface scale resolved", …)`：增 `detected_F` / `applied_F(整数策略后)` / `F_canvas` / `item_layer_scale` / `item_layer_pos_physical` / `current_screen`。
