## 二、关键技术实现评价

### 2.1 色键抠图 (PFMatting) ★★★★★

**做得好的地方**：
1. **背景推断策略严谨**：不仅看8个采样点，还统计整条边界覆盖率（< 60%则报非纯色底），有效降低渐变底误删风险
2. **两种模式设计合理**：
   - `flood`模式保留物体内部同色区（如白色高光）是正确的默认选择
   - `global`模式处理镂空场景
3. **容差0精确路径**：避免OKLab转换开销，直接RGBA字节精确BFS
4. **边缘处理细致**：泛洪边界1px内做半透明判定

**可改进点**：
- [ ] **性能优化空间**：当前对高分图仅"警告建议先清洗"，可考虑：
  - 自动降采样执行抠图后再上采样mask（类似M1清洗的分辨率策略）
  - 或提供"快速预览"模式（低分辨率计算+实时预览）
- [ ] **边缘羽化参数暴露**：当前半透明判定逻辑在代码中，UI未暴露羽化范围控制
- [ ] **非纯色底处理**：返回warning后用户体验断裂，可提供：
  - 自动降级到"魔棒多次点击"的半自动模式
  - 或显示可信度热力图让用户决策

### 2.2 选区模型 (PFSelection) ★★★★☆

**做得好的地方**：
1. **数据结构合理**：PackedByteArray + bbox缓存平衡了内存与性能
2. **布尔运算完整**：并/差/交三种操作，支持Shift/Alt快捷键语义
3. **性能达标**：256×256魔棒实测26.39ms < 50ms预算

**可改进点**：
- [ ] **魔棒容差调整体验**：报告提到"滚轮微调实时更新"，但当前UI未实现，后续M6需补齐
- [ ] **选区序列化**：未见选区保存/加载能力，用户关闭工具后选区丢失
- [ ] **选区变换**：缺少移动/缩放/旋转选区的能力（常见需求：微调选区位置）
- [ ] **反选功能**：布尔运算缺少`invert()`全图反选

### 2.3 连通域切分 (PFSegmenter) ★★★★★

**做得好的地方**：
1. **merge_distance启发式设计优秀**：解决了"飘在头顶的火焰"等实际痛点
2. **bbox更新用返回值**：避免了值类型传参陷阱，代码健壮
3. **噪点过滤**：min_area默认4px过滤碎屑
4. **排序规范**：top-left栅格顺序输出，命名可预测

**可改进点**：
- [ ] **合并策略增强**：
  - 当前merge_distance是全局固定值，可考虑自适应（如根据组件大小比例）
  - 提供"手动框选合并/拆分"功能（计划中提到但未实现）
- [ ] **预览交互**：报告提到"UI实时预览切分框，用户可调距离"，当前仅批量执行，缺少实时预览
- [ ] **切分结果命名**：`{原名}_01..` 序号命名较简单，可提供：
  - 按bbox位置命名（如`left_top`, `center`）
  - 或让用户自定义命名模板
- [ ] **重叠处理**：未说明如何处理重叠组件（虽然透明背景下少见）

### 2.4 描边工具 (PFOutliner) ★★★★☆

**做得好的地方**：
1. **形态学算法正确**：膨胀/腐蚀实现外/内描边
2. **颜色感知移除**：使用OKLab距离而非简单腐蚀，避免误删sprite边缘
3. **选择性描边**：selective mask支持（如仅下半部描边）
4. **往返测试**：加描边→移除 IoU ≥ 95%

**可改进点**：
- [ ] **remove_outline局限性**：
  - 标注为beta且有损，实际应用场景可能受限
  - 建议增加"智能检测"：分析颜色直方图判断是否存在描边层
  - 提供置信度指标（如"该图85%可能有黑色描边"）
- [ ] **描边宽度**：当前仅支持1px，实际需求可能要2-3px描边
  - 核可扩展为5×5、7×7
  - 或支持多次迭代膨胀
- [ ] **渐变描边**：彩色描边仅"加深30%"，可提供更多样式：
  - 高光描边（变亮）
  - 渐变过渡（从深到浅）
- [ ] **描边预览**：报告提到"提供预览"，但未明确UI形态，建议实时对比视图

### 2.5 导出器 (PFExporter) ★★★★★

**做得好的地方**：
1. **格式支持完整**：单PNG、批量PNG、spritesheet + JSON
2. **标准兼容**：TexturePacker JSON hash格式，与Godot `AtlasTexture`兼容
3. **放大导出正确**：最近邻插值保证像素锐利（颜色集合不增）
4. **性能达标**：100素材 < 10s

**可改进点**：
- [ ] **spritesheet打包算法**：
  - 未说明使用何种装箱算法（简单行列 / shelf / maxrects？）
  - 空间利用率未测试（建议补充"100个不规则素材的填充率≥X%"指标）
- [ ] **JSON格式扩展**：
  - 当前是TexturePacker "hash"子集，可考虑支持"array"格式（某些引擎偏好）
  - 添加pivot point字段（角色动画常用）
- [ ] **导出选项不足**：
  - 缺少压缩级别控制（PNG可以选择速度/体积平衡）
  - 未支持WebP等现代格式
  - 放大导出仅最近邻，可增加EPX/Scale2x算法（像素画专用）
- [ ] **批量命名**：
  - 当前`{name}.png`模板简单，可提供变量：`{index}`、`{tag}`、`{size}`等
  - 支持名称冲突处理策略（覆盖/重命名/跳过）
- [ ] **导出进度**：100素材10s期间UI是否冻结？应走task_queue异步+进度条

## 三、架构与代码质量评价

### 3.1 分层架构 ★★★★★

**优秀实践**：
- `core/pixel/` 纯算法，无UI依赖
- `services/` 组合core能力，提供业务接口
- `ui/shell/m2_action_controller.gd` 持有画布/检查器引用，正确放在UI层
- `main.gd` 仅委派，保持薄控制器模式

**这是教科书级别的分层**，为后续M3节点化预留了极好的重构基础。

### 3.2 测试策略 ★★★★★

**覆盖全面**：
- 单元测试：每个core模块独立fixture验证
- 集成测试：完整pipeline（白底→抠图→切分→描边→入库→spritesheet）
- 性能测试：魔棒26.39ms、批量清洗252.26ms、总耗时780ms全部达标
- 出口门控：`verify_m2.sh` 一键验证

**可补充**：
- [ ] **边界case测试**：
  - 空图、1×1图、巨大图（8192×8192）
  - 全透明图、无透明图
  - 极端颜色（纯黑/纯白/灰度）
- [ ] **异常路径**：
  - 文件写入失败（磁盘满）
  - 内存不足（大图处理）
- [ ] **回归测试**：固定几张"黄金样本"图，记录其抠图/切分结果的hash，防止算法改动意外改变输出

### 3.3 性能预算管理 ★★★★☆

**做得好**：
- 明确量化指标：魔棒<50ms、批量清洗<300ms（relaxed）
- 实测数据记录在案：26.39ms、252.26ms
- 提供strict模式（100ms）用于复核

**改进点**：
- [ ] **性能衰退检测**：当前手动跑测试，可集成到CI自动比对历史基准
- [ ] **大图性能**：仅对256×256测试，应补充1024×1024、4096×4096场景
- [ ] **内存监控**：未见内存占用预算，大批量处理可能OOM

## 四、用户体验评价

### 4.1 已实现的UX ★★★☆☆

**当前状态**：
- 顶栏三个按钮：Matte / Slice / Outline
- 批量操作：选中素材→点击按钮→异步执行→结果入库

**体验问题**：
1. **无实时预览**：用户不知道操作结果，需执行后才能看
2. **参数调整困难**：容差、merge_distance等参数如何暴露？
3. **撤销能力弱**：批量操作后生成新素材，原素材保留，但用户可能想"撤销重试"
4. **错误反馈不足**：报告提到"非纯色底"返回warning，但UI如何呈现？

### 4.2 交互设计建议

#### 4.2.1 抠图工具改进

**建议实现分步骤交互**：
```
用户点击"Matte"按钮
  ↓
弹出参数面板：
  - 背景色：[自动检测] [手动拾取]
  - 模式：○ Flood（保留内部同色）  ○ Global（全局色键）
  - 容差：[滑块 0-100]  实时预览
  - 边缘：☑ 半透明过渡  ○ 二值化
  ↓
实时预览区：左原图 | 右抠图结果（棋盘背景）
  ↓
[应用] [取消]
```

**关键改进**：
- 实时预览（<=100ms延迟，可降采样预览）
- 可视化背景色检测（高亮8个采样点，显示覆盖率%）
- 失败时给出建议（"背景不纯，建议用魔棒手动选择"）

#### 4.2.2 切分工具改进

**当前痛点**：merge_distance参数隐藏，用户无法调整

**建议**：
```
点击"Slice"按钮
  ↓
画布进入预览模式：
  - 所有连通域用彩色bbox框标识
  - 右侧面板：
    * 合并距离：[滑块 0-10px]  实时更新框
    * 最小面积：[输入框 默认4px²]
    * [手动合并]：点击两个框连线合并
    * [手动拆分]：框内画线切割
  ↓
检测到6个组件，列表显示：
  ☑ Component_01  (128×64px)
  ☑ Component_02  (32×32px)
  ...
  ↓
[提取选中的N个] [取消]
```

#### 4.2.3 描边工具改进

**建议增加样式预设**：
```
点击"Outline"按钮
  ↓
描边样式：
  ○ 无描边
  ○ 黑色外描边 1px
  ○ 白色外描边 1px
  ○ 彩色外描边（色相加深）
  ○ 自定义... [颜色选择器] [宽度1-3px] [内/外]
  ↓
☑ 选择性描边（仅下半部）
  ↓
实时预览
  ↓
[应用] [取消]
```

## 五、未完成项评估

### 5.1 跨平台验证延期 - 风险：中

**影响**：
- macOS Retina：字体/按钮可能过小
- Windows 150%：可能模糊或布局错乱

**建议**：
- [ ] 优先级：**高**，应在对外alpha前完成
- [ ] 录屏测试：三种环境各录1分钟操作视频
- [ ] 补充自动化：smoke测试增加截图对比（固定DPI下UI快照hash）

### 5.2 魔棒/矩形/套索UI未实现 - 影响：大

**当前状态**：core能力已完成，但无交互UI

**影响用户体验的场景**：
1. 用户无法"点击画布某处魔棒选择"（仅能批量Matte）
2. 无法用套索工具圈出不规则区域
3. 无法组合使用（先魔棒后减去矩形）

**建议**：
- [ ] 优先级：**高**，这是"手动选区抠图"的核心体验
- [ ] 实现路径：
  - M2.1补丁：快速实现基础交互（画布OverlayLayer + 工具状态机）
  - 或延后到M6编辑器：与其他编辑工具统一设计

### 5.3 content_scale_factor迁移延期 - 风险：低

**决策合理**：维持自研ui_scale体系，避免M2期间引入缩放回归风险

**技术债管理**：
- [ ] 建议立即创建独立卡片："GTK-缩放体系迁移评估"
- [ ] 在M3或M4开工前完成评估
- [ ] 若迁移，需完整回归：所有UI组件、FileDialog、画布fractional scale

## 七、对标业界产品

### 7.1 与Aseprite对比

| 功能 | PixelForge M2 | Aseprite | 差距 |
|------|---------------|----------|------|
| 魔棒选区 | ✅ core完成 | ✅ 成熟UI | 缺交互UI |
| 色键抠图 | ✅ flood/global双模式 | ✅ 仅global | **我们更好** |
| 连通域切分 | ✅ 自动+merge启发 | ❌ 无 | **我们独有** |
| 描边工具 | ✅ 外/内/彩色 | ✅ 类似 | 平齐 |
| spritesheet导出 | ✅ PNG+JSON | ✅ 多格式 | 基本满足 |

**结论**：核心能力已达到甚至超越Aseprite，主要差距在UI完成度。

### 7.2 与Photoshop对比

| 功能 | PixelForge M2 | Photoshop | 定位差异 |
|------|---------------|-----------|----------|
| 魔棒容差 | 0-100 OKLab | 0-255 RGB | 我们感知更准 |
| 选区羽化 | 硬边（1px半透明） | 可调羽化半径 | 我们简化（像素画够用） |
| 智能选择 | flood启发式 | AI边缘检测 | PS更智能，我们够用 |
| 批量处理 | ✅ 异步队列 | ✅ 动作录制 | 平齐 |

**结论**：我们不追求PS的AI能力，专注像素画场景已足够。

## 九、总结

### 9.1 核心优势
1. **算法质量高**：flood抠图、merge_distance切分是亮点创新
2. **代码工程化**：分层清晰、测试完备、性能达标
3. **为节点化铺路**：core层已完全解耦，M3可直接封装

### 9.2 主要短板
1. **UI完成度**：算法100%，交互UI仅30%
2. **跨平台验证**：缺少实机测试
3. **用户体验**：缺少实时预览、参数调整、错误引导

### 9.3 推荐行动
- **发布策略**：当前状态不建议对外alpha，补完"立即执行"清单后再发
- **团队分工**：算法工程师已超额完成，需UI工程师补齐交互
- **迭代节奏**：M2.1补丁（1周）→ 内部dogfood（1周）→ 小范围alpha → M3开工

**最终评分**：技术实现9/10，用户体验6/10，综合8.5/10。补齐UI后可达9/10。
