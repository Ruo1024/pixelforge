## 2. 背景与决策翻转

**自研体系的来历**：M0 为修 macOS Retina 下界面缩小一半，建了"启动读 `DisplayServer.screen_get_scale()` → `_ui_scale` → 各组件自觉 `_scaled_int()`"的约定式缩放。

**结构性弱点（实锤）**：`_scaled_int` 散在 **7 个文件、68 处**（`main.gd` 17 / `cleanup_inspector.gd` 17 / `canvas_zoom_control.gd` 12 / 三个 dialog 各 6 / `m2_1_ui_controller.gd` 4），但 23 个 UI 文件里**只有 7 个接了 `ui_scale` 注入**——剩下 16 个是"无人区"。任何新组件忘记接线都不会被 lint/测试发现（`cleanup_inspector.gd` 当初就是这么让 mac 缩放复发的）。

**为什么 M2 当时选"维持自研"**：M2 新增可见 UI 仅顶栏按钮，复用了已缩放的尺寸；迁移需真机 + 原生 FileDialog + 画布 fractional 联合走查，改动面超出 M2 主线，故卡片结论是"建议后续单独立卡"。

**为什么现在翻转**：(1) M3 即将新增一大批画布原生 Control，**先换地基最省**；(2) 自研体系的"无人区"风险会随 UI 增多线性放大；(3) 最值钱的检测启发式可原样保留，迁移主要是"换应用方式 + 删约定 + 加守护"，收益/风险比好。**本卡即把决策卡结论从"维持自研"翻为"迁移"。**

## 3. 核心设计：方案2（chrome 继承 + 画布反向补偿 + 整数对齐）

### 3.1 现状机制（事实）

- `main.gd::_apply_viewport_scale_policy()`（~L222）**显式禁用**原生缩放：`content_scale_mode = DISABLED`、`content_scale_factor = 1.0`、`content_scale_stretch = FRACTIONAL`。
- 缩放全靠 `_ui_scale`（`_resolve_interface_scale()` 算）× 各处 `_scaled_int()`。
- 画布 `infinite_canvas.gd` 声明了 `ui_scale`（L34）但**绘图根本没用它**：美术内容靠 `camera_zoom` + `world_to_screen()` 绘制，`texture_filter = NEAREST`（L57/61/482）保像素硬边。Control 自绘的背景 / 像素网格 / 选区框 / 蚂蚁线在 **Control 空间**（用 `world_to_screen`），sprite 是 **item_layer 子节点**——已经是两套坐标系。

### 3.2 目标语义

- **chrome（面板 / 按钮 / 标签 / 对话框 / 弹窗）**：交给 `content_scale_factor = F` 自动继承缩放，**这是迁移的全部收益来源**——以后任何新 Control 自动正确，无需接线。
- **画布美术**：保持**像素精确**，不被 `F` 二次放大成糊块。把"100% 缩放"定义为 **1 美术像素 = `F_canvas` 物理像素**（显示器感知：Retina 上不再显得很小），且**美术像素在任何缩放/任何 F 下都占整数个物理像素**（杜绝 fractional 1.5x 接缝）。

### 3.3 不变量（可测合同）

```
F        = content_scale_factor（来自现有检测，可为 1.0 / 1.5 / 2.0）
F_canvas = max(1, round(F))            # 画布设备倍率，整数（1.5 → 2）
on-screen 美术放大（物理px/美术px）  = camera_zoom × F_canvas
```

当 `camera_zoom ≥ 1` 且为整数档时，`camera_zoom × F_canvas` 为整数 → NEAREST 下逐像素硬边、无接缝。`camera_zoom < 1`（缩小看全图）属下采样，允许非整数（NEAREST 下采样可接受）。**这条不变量是 §5 自动测试与人工走查的核心断言。**

### 3.4 推荐实现（Wiring A）+ 必须在 spike 验证的点

**净变换恒等式**：Godot 在视口级对所有 CanvasItem 乘 `F`。若把 `item_layer.scale = Vector2(F_canvas/F, F_canvas/F)`，则美术净放大 = `(F_canvas/F) × F × camera_zoom = F_canvas × camera_zoom`（与 `F` 解耦、整数对齐）。三个关键档位自洽：

| F (chrome) | F_canvas | item_layer.scale | 净美术放大 |
|---|---|---|---|
| 1.0 | 1 | 1.000 | camera_zoom（与今天一致）|
| 1.5 | 2 | 1.333 | 2 × camera_zoom（整数，无接缝）|
| 2.0 | 2 | 1.000 | 2 × camera_zoom |

> Godot 把 `item_layer.scale` 与 `F` 复合成**单个变换矩阵**，故中间的 1.333 不产生中间位图、净倍率精确为整数——但 `FRACTIONAL` stretch + NEAREST 的复合行为正是原决策卡的验证点 ②，**必须在 Phase 0 真机确认**。

**坐标系一致性清单**（item_layer 缩放后，下列每一处都要换算到同一净倍率，否则 sprite 与网格 / 选区 / 蚂蚁线错位）：

- `world_to_screen()` / `screen_to_world()`（输入命中、相机居中都依赖它）
- `_draw_pixel_grid()`（Control 空间，必须与 sprite 像素对齐）
- 选区 bbox / 蚂蚁线（`base_tool._draw_selection_bbox` 等）
- 批次卡放置 `_add_batch_card`、缩略图
- **zoom-to-cursor 锚点**（`set_camera_zoom` / `zoom_by_steps` 的 `screen_anchor`）
- 左下缩放 overlay（`canvas_zoom_overlay_controller` / `canvas_zoom_control`，是独立顶层 Control，受 `F` 但不受 item_layer.scale，对齐要单独核）
- 输入事件按逻辑坐标（pre-F）上报：item-local = 逻辑坐标需折算 `F/F_canvas`

**Wiring B（备选，spike 若发现 A 坐标系太绕则回退）**：不缩放 item_layer，改为把"净倍率 `F_canvas × camera_zoom`"直接烘进 sprite 放置与 `world_to_screen`。代码改动点更多但坐标系单一。Phase 0 二选一并写入本卡。

**边界**：`camera_zoom × F_canvas` 向下溢出到 0 时 clamp 到 1。

### 3.5 触点清单（迁移会改到的文件）

- `ui/shell/main.gd`：`_apply_viewport_scale_policy`（factor 改 = F）、`_build_app_theme`（字号去 `_scaled_int`）、`_apply_window_defaults`（窗口尺寸 / min_size 数学，删 mac 点-像素来回换算 ~L304-313）、删 `_scaled_int/_scaled_vec2*` 辅助、删 `_canvas.ui_scale=`/`_cleanup_inspector.ui_scale=` 等 7 处注入。
- `ui/canvas/infinite_canvas.gd`：§3.4 画布补偿（重点）。
- `ui/inspector/cleanup_inspector.gd`、`ui/shell/m2_1_ui_controller.gd`、`ui/canvas/canvas_zoom_control.gd`、`ui/dialogs/{matte,slice,outline}_dialog.gd`：删 `_scaled_int` 与 `ui_scale` 字段，尺寸/字号回归裸逻辑常量。
- `ui/shell/canvas_zoom_overlay_controller.gd`：去掉 `ui_scale` 传参。

## 5. 调试与测试参考方案

### 5.1 Headless 自动化测试（CI / agent 可跑）

为可测性，把纯逻辑从副作用里抽出来做静态/纯函数：

- `compute_canvas_device_scale(F: float) -> int`：断言 `1.0→1, 1.49→1, 1.5→2, 2.0→2`。
- `effective_art_pixel_px(camera_zoom: float, F: float) -> int`：断言不变量 `== round(camera_zoom × compute_canvas_device_scale(F))`，且 `>= 1`（含 zoom<1 的 clamp）。
- 视口策略：把 `_apply_viewport_scale_policy` 改成接受/读取注入 scale，测 `content_scale_factor == 注入值`、`content_scale_mode == DISABLED`。
- 保留现有检测族单测（`test_main_window_ui.gd` 的 4 个 `compute_auto_*`/`fit_*` 测）。
- 逻辑尺寸不回归：min-size / 顶栏 / 按钮断言保持。
- 新增文件建议：`tests/smoke/test_content_scale_policy.gd`、`tests/unit/test_canvas_device_scale.gd`。

### 5.2 静态守护脚本（本卡灵魂，替代无法做的"约定 lint"）

`pixel/scripts/check_ui_scaling.sh`（命中即非零退出）：

```bash
#!/usr/bin/env bash
set -euo pipefail
cd "$(dirname "$0")/.."
fail=0
# 1) 退役的辅助函数不得再现
grep -rnE "_scaled_int|_scaled_vec2" ui --include=*.gd && fail=1 || true
# 2) 手动 ui_scale 注入不得再现（画布补偿用新命名，见白名单）
grep -rn "ui_scale" ui --include=*.gd | grep -v "# scale-exempt" && fail=1 || true
# 3) custom_minimum_size 不得硬编码数字字面量（应来自逻辑常量）
grep -rnE "custom_minimum_size\s*=\s*Vector2i?\(\s*[0-9]" ui --include=*.gd && fail=1 || true
# 4) 字号 override 不得用数字字面量（应来自主题）
grep -rnE "add_theme_font_size_override\([^,]+,\s*[0-9]" ui --include=*.gd && fail=1 || true
[ "$fail" = 0 ] && echo "check_ui_scaling: ok" || { echo "banned scaling pattern found" >&2; exit 1; }
```

- 合法例外用行尾 `# scale-exempt: <理由>` 显式放行（脚本 grep -v 已留口）。
- `pixel/scripts/verify_m2_3.sh` = `lint.sh` → `run_tests.sh` → `check_ui_scaling.sh` → `check_export_templates.sh` → 暂存图片红线检查（沿用 `verify_m2.sh` 那段）。

### 5.5 接缝 / 双重缩放的量化判定

开已知 sprite，zoom=100%，截图后量像素块：每块须是**均匀的整数 `F_canvas` 物理像素**，不得出现宽度 `F_canvas±1` 的行/列（那是 fractional 漏对齐）。Win 150%（chrome F=1.5、F_canvas=2）下尤其要看 chrome 与画布能否共存——chrome 1.5x、画布 2x，二者在同屏不冲突即达标。

## 8. 显式延后项（诚实工程登记）

- **运行时动态重标定 / live re-scale**：拖滑条 / 跨显示器拖窗即时重算 UI，不重启。需监听 `window_changed` 类信号 + 重建主题/重算 F_canvas。本轮不做。
- **真·响应式布局**：窗口断点、面板 reflow / 可停靠 / 小窗自动折叠侧栏。属布局结构改造，与缩放正交，本轮不做。
- **多显示器运行时重算**：本轮只保证"启动按当前屏检测正确"，跨屏后不自动重标定。

## 9. 规范改动草案（提议待批准，随 Phase 4 落）

### 9.1 `AGENTS.md` §工程约定（保持精瘦的一行）

- 旧：`- 新 UI 组件必须接 ui_scale（\`_scaled_int()\`），禁止硬编码像素值。`
- 新：`- UI 缩放统一由 \`Window.content_scale_factor\` 继承，**禁止 \`_scaled_int()\` / 手动 \`ui_scale\` 注入 / 硬编码像素字号**；画布美术按"反向补偿 + 设备像素整数对齐"自管（规范本体见 \`ARCHITECTURE.md §5\`，守护见 \`scripts/check_ui_scaling.sh\`）。`

### 9.2 `ARCHITECTURE.md §5 编码规范`（新增 spec 本体一条）

> - **UI 缩放**：界面缩放由 `Window.content_scale_factor`（启动按 `_resolve_interface_scale()` 检测）统一驱动，所有 Control 尺寸/字号写**逻辑常量**、由 factor 等比放大，禁止 `_scaled_int()` 与组件级 `ui_scale` 注入。画布美术不随 chrome 二次放大：设备倍率 `F_canvas = max(1, round(F))`，净美术放大 = `camera_zoom × F_canvas`（整数对齐、NEAREST 硬边）。新增 UI 不需要任何缩放接线；`scripts/check_ui_scaling.sh` 做静态守护，合法例外用 `# scale-exempt:` 放行。

### 9.3 落地时机

**不可提前**。Phase 1-3 期间代码与旧规范暂时并存属正常迁移态；Phase 4 代码删净 `_scaled_int` 后，规范与守护脚本同一轮落，保证任一时刻"规范 = 代码现状"。
