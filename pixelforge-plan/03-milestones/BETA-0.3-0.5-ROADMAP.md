# Beta 0.3–0.5 原型冲刺总路线

> 计划日期：2026-07-12
>
> 前置基线：Beta 0.2 工程候选，项目所有者人工验收尚未执行
>
> 产品依据：[无限画布 90% 原型差距评估](../04-research/INFINITE-CANVAS-90-PARITY-ASSESSMENT.md)
>
> 执行书：[Beta 0.3](BETA-0.3-PLAN.md) → [Beta 0.4](BETA-0.4-PLAN.md) → [Beta 0.5](BETA-0.5-PLAN.md)

## 1. 这三轮只完成一件事

把 PixelForge 做成一个图像向的无限画布原型：用户从空画布加入提示词和参考图，在画布中组织生成分支，调用云端模型得到候选，把结果继续用于下一轮，再把成熟的一组流程保存成可复用工作流。

三轮全部完成后，图像向 LibTV 无限画布子集达到至少 90% 的工程候选，随后冻结原型功能并进入打磨。Beta 0.3 或 Beta 0.4 单独完成时都不构建对外候选，也不请求项目所有者分段验收。

```text
Beta 0.2 工程基线
   └─> Beta 0.3 内容工作区
          └─> Beta 0.4 云端生成与结果回流
                 └─> Beta 0.5 组运行与工作流复用
                        └─> 唯一候选包 + 项目所有者统一人工验收
                               ├─ go：允许后再合并 main / push
                               └─ 不通过：留在未推送分支修复或等待放弃决定
```

## 2. 已锁定的产品边界

### 三轮必须完成

- 一张画布容纳提示词、结构化批量输入、参考图、风格、生成、候选结果和阶段组；
- 从空白处创建，从系统拖入或粘贴图片后在落点直接成卡；
- 复制、粘贴、删除、撤销、连线、分组和大画布导航；
- 选择具体云端模型，并只显示模型实际支持的设置；
- 一张或多张参考图真正进入云端请求；
- 任务状态、失败原因、取消、重试、费用和结果来源在画布原位可见；
- 结果可以直接成为下一轮参考或新分支，不需要先导出再导入；
- 结构化批量输入、整组运行和本地工作流模板；
- 保存重开、自动保存、Undo/Redo、双语、UI 缩放和结果溯源继续有效。

### 永久排除，不登记为待开发债务

- 账号、登录注册、角色和权限；
- 多人协作、评论、共享链接和在线发布；
- 项目版本历史、快照时间线和历史版本回滚；
- 云同步、社区、市场和排行榜。

Undo/Redo、自动保存、崩溃恢复和生成结果来源不属于“版本历史”，仍然是数据安全核心。

### 无限期延后

- 本地模型和 ComfyUI 产品接入；
- 视频、音频、配音和时间轴；
- Agent 自动建图或自动制片；
- 独立 LLM 对话节点。

已有本地模型/ComfyUI 底层代码不在本轮删除，但必须退出主界面和主添加入口，也不得继续扩展。

### 原型后再决定

- 嵌套组、组折叠、卡片自由缩放、锁定、自动排布；
- 搜索节点、书签、视口历史和复杂导航；
- 持久任务中心、通用脚本语言和任意循环节点；
- 携带素材的工作流压缩包、跨项目剪贴板；
- 大量云端服务商、模型市场和第二套智能画布；
- 万级节点优化、通用布局引擎和框架级重构。

## 3. 90% 的固定验收尺

最终计分不按代码量，也不因 PixelForge 自创功能加分。

逐项定义、唯一主卡和最终场景锁定在 [90% 受控矩阵](BETA-0.3-0.5-PARITY-MATRIX.md)。下面只保留总数和范围摘要。

- 48 个核心项，每项 2 分，共 96 分；
- 18 个次要项，每项 1 分，共 18 分；
- 总分 114；
- 48 个核心项必须全部工程完成，再完成至少 7 个次要项，才达到 103/114。

三轮计划覆盖全部核心项，并预留 12 个次要项作为余量：

| 编号 | 本轮目标 |
|---|---|
| W06 | 项目级入口与最近项目 |
| W07 | 快速起步模板 |
| M07 | 卡片展开编辑 |
| C07 | 批量条目编辑 |
| C08 | 参考素材替换/删除 |
| L07 | 连线显隐 |
| N05 | 小地图 |
| G10 | 风格库与检索 |
| G12 | 复用 PixelForge 现有图像加工链 |
| S03 | 把组保存为工作流 |
| S04 | 插入后继续编辑和运行 |
| Q03 | 常驻控件不遮挡主要内容 |

明确不计入计划的六个次要项为：公开只读/复制（W08）、跨画布复制（L08）、供应商余额/积分入口（R08）、焦点生成编辑（G11）、跨项目流程复制（S05）、分享只读画布（S06）。单次请求费用仍作为 PixelForge 运行反馈保留，但不冒充 R08。上述排除不影响达到 90%。

无论分数多少，下面四项失败时都不能称为原型完成：

1. 中英文存在文字重叠、截断或不可读字形；
2. 常见 macOS 窗口尺寸下关键操作被裁切；
3. 空画布找不到第一步；
4. 选择、连接、运行、失败和完成没有明确反馈。

## 4. 开发节奏：一个 Goal 连续完成三轮

项目所有者确认本计划后，下达一次开始开发/Goal 指令即视为三版执行授权。执行 agent 从 Beta 0.3 连续推进到 Beta 0.5，不因单张卡、单个组件或单个版本停下等待用户。只有以下情况允许暂停：

- 需要改变本路线已经锁定的产品含义；
- 需要超出下文预批准边界修改跨模块契约；
- 自动化红灯无法在当前范围内排除；
- 发现会造成数据丢失、密钥泄漏或许可证违规的风险。

每张卡仍是边界清晰的实现单元，但提交和报告按整体切片收口：

| 集成边界 | 允许的验证 | 记录方式 |
|---|---|---|
| 卡内开发 | 与本次改动直接相关的定向自动化 | 不单独写体验报告 |
| Beta 0.3 出口 | 全量自动化 + 脚本截图 | 本地提交与测试日志；自动进入 Beta 0.4 |
| Beta 0.4 出口 | 全量自动化 + 脚本截图 | 本地提交与测试日志；自动进入 Beta 0.5 |
| Beta 0.5 出口 | 全量自动化 + 首次统一候选构建 + 脚本截图 | 一次性完成三版工程报告与人工验收单 |
| 三轮之后 | 项目所有者真实云端、完整旅程和真机人工验收 | go / 修复后复验 / 不合并 |

统一工程报告使用 `reports/BETA_0_3_TO_0_5_ENGINEERING_REPORT.md`，只在 Beta 0.5 收口时创建。`CURRENT-STATE.md` 也只在三版全部达到工程出口或真正阻塞时更新，不为单卡、Beta 0.3 或 Beta 0.4 反复改状态。

## 5. 测试纪律：开发中完全不使用 Computer Use

Beta 0.3–0.5 开发期间：

- 不使用 Computer Use 点击、输入或跑完整工作流；
- 不做逐卡、逐组件或逐版本的 agent 交互式冒烟；
- 不请求项目所有者分段人工测试；
- 不调用真实付费云端服务；
- 只运行单元、集成、UI 场景、项目往返、云端录制响应和静态守护脚本；
- UI 改动只允许由自动脚本生成固定语言、窗口尺寸和画布倍率的截图，用于发现明显空白、重叠、裁切或状态错误。

脚本截图是工程证据，不等于体验通过。真实云端、输入手感、视觉质量、跨显示器和完整创作旅程统一留给 Beta 0.5 后的项目所有者人工验收。

每版出口至少运行：

```bash
./pixel/scripts/lint.sh
./pixel/scripts/run_tests.sh
./pixel/scripts/check_i18n_catalogs.sh
./pixel/scripts/check_ui_scaling.sh
./pixel/scripts/check_export_templates.sh
git diff --check
```

执行阶段可新增一个统一的 Beta 验证脚本和自动截图脚本，但不得为每张卡复制一套验证脚本。

## 6. 外部项目只在模糊点提供行为证据

参考项目固定为 [hero8152/Infinite-Canvas](https://github.com/hero8152/Infinite-Canvas/tree/main)。当前审计快照为 `469b41b`；若上游变化，子 agent 必须记录实际查看的提交。

只有任务卡已标注的交互仍然模糊时，执行 agent 才开启一个低上下文只读子 agent：

1. 只查对应模块，不通读整个仓库；
2. 只回传“入口 → 可见状态 → 反馈 → 结果 → 异常”的行为表；
3. 标出相关文件或函数位置，但不粘贴大段源码；
4. 不复制代码、数据结构、文案、布局、CSS、截图、资产或工作流文件；
5. 主 agent 根据 PixelForge 产品原则和契约独立做最终设计；
6. 一旦行为已经明确，立即回到开发，不继续扩张调研。

优先参考范围：

| PixelForge 卡 | 外部项目只读参考主题 |
|---|---|
| B3-1 / B3-4 | 空白创建、拖放/粘贴、多图排布、复制内部连线、分组 |
| B3-5 | 小地图、适配全部、低倍率结构 |
| B4-3 / B4-5 | 任务占位、失败/完成状态、输出转输入 |
| B4-6 | 项目素材与风格的搜索/插入 |
| B5-1 / B5-3 | 级联运行、组选区保存和工作流再次插入 |

外部仓库许可证禁止未经授权的商业封装，并要求二次开发保持开源署名；因此它只能作为行为证据，不能成为 PixelForge 的实现底座。

## 7. 计划批准后允许的最小契约补全

项目所有者批准本总路线和三份执行书后，视为原则上批准下面三组最小补全。执行 agent 应先提交明确的契约 diff，由主 agent 审核后再写代码；若含义超出本表，仍需重新请求项目所有者决定。

| 版本 | 允许补全 | 不允许顺带加入 |
|---|---|---|
| Beta 0.3 | `frame` 的 graph、标题、颜色和边界；成员唯一真相为节点 `frame_id`；卡片展开态 | frame 内另存 member_ids、跨 graph/嵌套/折叠组、锁定、自动布局、第二套画布 |
| Beta 0.4 | Provider 标准模型描述、请求顶层 `model_id`、有序多参考、精确生成快照与目标感知运行 | 新本地模型、ComfyUI、全局任务系统、UI 写死模型表、供应商私有第二套参数真相 |
| Beta 0.5 | 结构化输入行、组运行范围；白名单本地工作流模板的节点、内部连线和相对布局 | 项目版本历史、素材打包、共享、跨项目剪贴板、通用脚本/循环 |

旧项目缺少新增字段时必须使用安全默认值；保存重开和未知字段保留仍是硬门。

### 7.1 三版共用兼容矩阵

| 旧数据 | 新行为 | 保存要求 |
|---|---|---|
| `ai_generate` 没有 `model_id` | 传空值，由对应 provider 解析默认模型 | 不在 UI 猜一个全局默认 |
| 只有 `ref_image` / 单数参考 provenance | 作为有序列表第一个元素读取 | 新数据写复数字段；旧字段原文保留 |
| 没有 `frame_id` | 视为未分组 | 不自动猜空间归属 |
| `frame_id` 指向不存在的 frame | 以未分组方式显示并给结构化警告 | 失效 ID 原文保留 |
| 历史 `object_list.items` | 只读转换为稳定 rows | `rows` 存在后为权威；旧字段保留但不再执行 |
| 未知节点、参数或历史 Comfy 节点 | 幽灵/隐藏兼容，不执行 | open → save → reopen 不丢原文；禁止保存进工作流模板 |

每次契约补全都必须用同一兼容矩阵验证旧项目打开、保存和重开，不得在各卡重复发明迁移规则。

## 8. Git、候选和回退

- 在本地 `codex/` 集成分支开发；多 agent 改代码时使用独立 worktree；
- 每个完整集成切片保留可回退提交，不为纯机械小改制造大量提交；
- 不覆盖现有 `pixel/project.godot` 用户差异；
- commit 前检查 staged 文件和受保护图片；
- Beta 0.3、Beta 0.4 不构建候选包，不记录候选校验和；
- Beta 0.5 全部工程门通过后构建首次统一候选；项目所有者验收失败后的修复可以递增候选号重建；
- 项目所有者人工 `go` 前，不合并 `main`、不 push；
- 若人工候选不通过，优先在当前未推送分支修复；整体放弃或破坏性回退必须等待用户确认。

## 9. 三轮最终出口

三轮自动化完成时，只能写：

> Beta 0.3–0.5 工程通过；图像向无限画布 90% 工程矩阵达到候选门槛；项目所有者人工与发布均待验。

只有项目所有者完成统一人工清单并明确 `go` 后，才允许更新为“人工通过”，并另行决定是否合并和推送。
