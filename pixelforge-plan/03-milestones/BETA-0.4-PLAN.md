# Beta 0.4 — 云端生成与结果回流

> 状态：Beta 0.3 工程门通过后连续执行
>
> 总路线：[Beta 0.3–0.5 原型冲刺](BETA-0.3-0.5-ROADMAP.md)
>
> 目标区间：把图像向无限画布原型推进到约 80%–84%

## 1. 本轮用户旅程

> 用户在提示词、风格和参考素材旁选择一个具体云端模型。界面只显示该模型支持的尺寸、数量和参考图能力。点击运行后，结果位置立即出现占位和状态，画布仍可操作；完成后候选进入真实结果卡。用户查看模型、提示词、随机种子、参考来源和费用，选中一张“作为参考”或“基于此继续”，画布在原分支旁创建新分支。保存重开后，两轮结果和来源关系都保持。

本轮只做云端图片模型。开发自动化使用本地模拟服务和录制响应，不调用真实付费 API；真实密钥统一留到 Beta 0.5 后由项目所有者验收。

## 2. 必做与明确不做

### 必做

- 用户选择具体模型，不只选择抽象服务商；
- 至少两个已有云端模型条目可选择，其中至少一个真实支持两张或更多参考图；
- 一张或多张参考图真实进入模型请求，不能只做 UI；
- 常用参数按模型能力显示，不支持时在运行前阻止；
- 原位占位、运行、失败、取消、重试和完成；
- 结果详情、候选审阅、加工/导出与结果回流；
- 现有项目素材快捷入口和风格检索。

### 明确不做

- 本地模型、ComfyUI 或新的本地执行路径；
- 大量服务商扩张、模型市场、在线提示词库；
- 视频、音频、LLM 和 Agent；
- variation、扩图、局部重绘、放大等模型未明确支持的假入口；
- 全局持久任务中心；
- Computer Use、真实付费调用或项目所有者分段测试。

## 3. 依赖与集成顺序

```text
B4-0 云模型能力与最小契约
   ├─> B4-1 生成卡常用设置
   └─> B4-2 多参考素材
           └─> B4-3 原位任务与结果占位
                    └─> B4-4 候选详情与溯源
                             ├─> B4-5 结果回流新分支
                             └─> B4-6 项目素材/风格快捷入口
                                      └─> B4-7 集成出口
```

B4-1 与 B4-2 可在 B4-0 契约稳定后并行。B4-3 至 B4-5 必须顺序合流，避免同时修改任务状态、结果物化和 provenance。

## 4. 执行卡

### B4-0 云模型能力与最小契约

**服务旅程**：让卡片在提交前知道模型真正能做什么，避免静默截断和运行后才报“不支持”。

**交付**：

- 执行时以当前官方文档和已有录制响应核对现有云端实现，优先扩展现有 OpenAI/RetroDiffusion 路径，不先增加新服务商；
- 至少暴露两个经过自动化契约验证的云端模型条目，其中至少一个能使用两张或更多参考图；
- 如果现有两个适配器均不能满足真实参考图核心旅程，只允许补一个最小云端适配器，不建设通用模型市场；
- Provider 契约新增标准 `get_model_descriptors()`；每项至少为 `{provider_id, model_id, display_name, is_default, capabilities}`，UI 与运行层只能读取该目录，不得写死第二份模型表；
- `PFGenRequest.model_id` 是顶层字段；旧节点使用空字符串，由所属 Provider 自己解析默认模型，不能由 UI 猜一个全局默认；
- 每个模型能力至少说明：文字生成、`max_reference_images`、允许尺寸/比例、批量上限、随机种子、透明背景和费用估算；
- `PFGenRequest.ref_images` 是有序图像列表，旧 `ref_image` 作为第一项兼容读取；`ai_generate` 参考输入扩为 `image_list`，单张 `image` 沿用现有自动包装；
- 新 provenance 写有序 `reference_asset_ids` 和内容哈希数组；旧单数字段继续兼容，引用扫描同时覆盖新节点的 live 引用和复数 history 引用；
- 每个结果保存经过清理的生成快照：provider/model、提示词、负面词、风格、尺寸、随机种子、实际参考图、`source_generate_node_id`、可选 `source_row_id` 和 `run_id`；绝不保存密钥、绝对路径或完整外部响应；
- 定义目标感知的运行协调边界：调用必须带 `graph_id + generate_node_id`，可带目标 `batch_node_id`；状态和结果按 run/目标节点记录，不能再依赖当前画布选择或“第一个同类型节点”；
- 所有模型私有字段仍通过既有服务边界，不在 UI 另建第二份逻辑真相。

**完成条件**：模型切换后卡片状态立即正确变化；旧项目可打开；不支持的数量、尺寸或参考图在提交前有明确原因。

**自动化**：模型描述符完整性、旧默认值、模型到录制请求映射、能力前置校验、项目往返和云端错误映射。

**决策边界**：若需要超出总路线授权增加新的契约语义，必须暂停；仅选择当前云端模型和更新录制 fixture 不需要项目所有者分段决定。

### B4-1 生成卡常用设置

**服务旅程**：用户在画布附近就能看懂本次会用哪个模型、生成什么、生成多少。

**交付**：

- 卡面显示服务商、具体模型、提示词来源、风格、画幅、目标像素尺寸、数量、随机种子、费用估算和运行入口；
- 明确区分“云端实际输出尺寸”和“PixelForge 最终目标像素尺寸”；
- 负面提示词、参考强度、透明背景等只在模型声明支持时显示；
- 模型切换后，数量上限、尺寸、参考图和费用同步更新；
- 批量上限不再出现 UI 允许 16、实际请求只发 4 的静默差异；
- 高频设置在卡面，低频服务专有设置放展开区；
- 折叠后仍能识别模型、数量和主要输入。

**完成条件**：用户无需运行即可理解请求；错误设置不能提交；修改参数后成本和摘要一致。

**自动化**：卡内编辑、能力显隐、上限、尺寸映射、成本刷新、Undo/Redo、保存重开、中英文长文几何和 i18n 裸字符串守护。

### B4-2 多参考素材模块

**服务旅程**：用户能把参考素材组织成有顺序的真实输入，并知道模型最终用了哪些图。

**交付**：

- 新增最小 `reference_set` 内容模块，持有有序 `asset_ids` 并输出 `image_list`；现有单图 `image_input` 保持兼容；
- 卡内显示真实缩略图、顺序、名称、来源和失效状态；
- 支持从系统、画布结果和项目素材库加入，支持排序、替换和移除；
- 超过模型上限时要求用户选择要使用的图，不自动丢弃尾部；
- 模型不支持参考图时说明原因，不能假装已经使用；
- 每次生成记录实际进入请求的参考素材和内容哈希；至少一个录制模型 fixture 必须证明两张参考图都进入真实请求映射；
- 参考图移除或 Undo 不隐式删除 AssetLibrary 素材。

**完成条件**：一张和多张参考都沿同一图输入路径进入录制请求；保存重开后顺序和来源不变；坏素材不阻止项目打开。

**自动化**：导入、排序、替换、删除、上限选择、Undo、项目/图 round-trip、损坏素材降级、引用扫描、录制请求顺序和 provenance。

### B4-3 原位任务占位与状态

**服务旅程**：点击运行后立即知道任务已提交，等待期间仍能继续使用画布。

**固定产品语义**：

- 第一次运行可以填充已经连接的空结果卡；没有结果卡时在生成卡右侧创建一个；
- 提交后立即显示预期数量和比例的占位格；
- 失败或取消的占位保留原因、重试和移除；重试填充原占位；
- 已经有成功结果后再次运行，创建新的相邻结果卡，不覆盖旧结果；
- 成功结果的“再次运行”同样创建新结果卡；
- 运行目标以按钮所属生成节点为准；
- 本卡实现 B4-0 的目标感知运行协调器；同一 graph 上两张生成卡可以分别运行、分别落到自己的结果卡和 run 状态，不得再扫描“第一个 Provider/Batch”；
- 等待、运行、取消中、失败和完成使用同一状态体系；没有可靠百分比时只显示真实阶段，不伪造进度。

**完成条件**：画布在任务期间仍可平移、缩放和编辑其他分支；失败给出下一步；取消或失败不破坏上一批稳定结果。

**自动化**：录制的等待/完成/失败/超时/取消/重试序列，占位出现时机，重复提交防护，旧结果不覆盖，TaskQueue 取消传播、同图双生成节点的目标隔离和状态截图。

**不做**：全局任务中心和长期任务历史。

**外部参考触发点**：占位与结果合并仍有歧义时，子 agent 只读外部项目的 pending/output 行为，只回传状态变化。

### B4-4 候选审阅、详情与结果来源

**服务旅程**：用户能比较候选，并知道某张结果来自什么设置。

**交付**：

- 复用现有宫格、聚焦、keep/reject/flag、过滤和前后对比；
- 为单个候选提供详情：提示词、模型、随机种子、尺寸、参考图、费用、生成时间和来源分支；
- 详情与“复制设置”只读取结果生成当时的安全快照，不回读后来可能已经修改的生成节点，也不再复制一份 UI 元数据；
- 提供复制提示词、复制设置、再次运行、进入现有清洗/抠图/描边/像素编辑和导出；
- 不显示无意义的空字段，不记录密钥或完整外部响应；
- 批次中每个结果必须对应正确的输入条目和参考来源。

**完成条件**：点击候选即可理解来源；复制设置后可修改；现有审阅、加工和导出没有回归。

**自动化**：provenance 物化/往返、候选与随机种子/参考/费用对应、缺失旧元数据降级、审阅/加工/导出回归和详情 UI 映射。

### B4-5 结果成为下一轮输入

**服务旅程**：候选不再是工作流终点，用户无需导出再导入即可继续创作。

**固定动作**：

- “作为参考”：单图创建 `image_input`；多选创建 `reference_set`；
- “基于此继续”：创建参考模块、复制当前生成设置的生成卡和空结果卡，并连成新分支；
- 新分支出现在原结果右侧，原分支不移动、不修改；
- 新分支只准备设置，不自动发起付费请求；
- 新参考模块引用原结果 asset，原结果 provenance 保持；新的执行连线只从参考模块连接到新生成卡。结果卡与参考卡之间不制造无执行含义的假连线；
- 整个动作是一次 Undo。

**完成条件**：新分支可以立即修改提示词、模型或参数后运行；保存重开后原结果、新分支和来源都在。

**自动化**：单图/多图回流、新节点和内部连线、ID、位置、原子 Undo/Redo、项目往返、原结果不覆盖和来源缺失错误。

**不做**：模型未声明支持的 variation、扩图、重绘或放大按钮。

**外部参考触发点**：只在“转换为输入”和“复制为输入”仍不清楚时参考外部项目；PixelForge 最终固定采用不破坏原分支的新分支语义。

### B4-6 项目素材与风格快捷入口

**服务旅程**：用户可以在画布旁找到已有项目素材和风格并直接放回创作空间。

**交付**：

- 复用现有资源区域，提供“项目素材 / 风格”两类，不新建通用资源管理系统；
- 项目素材复用现有分类/搜索能力，风格支持名称和类别检索；
- 项目素材复用 AssetLibrary，拖入空白处创建参考卡；
- 风格只复用现有 StylePreset，拖入分支时创建或替换风格来源；
- 空结果、损坏素材和模型不兼容均给出简单原因；
- 不增加提示词模板 CRUD、最近使用、账号或在线库；提示词继续在内容卡直接编辑。

**完成条件**：项目素材能进入画布，风格可搜索并进入生成分支；不另建素材或风格的第二套数据真相。

**自动化**：素材搜索/分类、风格检索、两类拖放、去重、坏素材降级、项目往返和中英文空状态截图。

**外部参考触发点**：只有素材“拖到空白”和风格“替换现有来源”的差异模糊时，只读参考外部素材库行为，不参考布局。

### B4-7 Beta 0.4 工程出口

**自动化旅程**：

1. 用提示词和两张参考素材建立分支；
2. 选择支持参考图的云端模型；
3. 通过本地录制响应完成等待、生成和多个候选；
4. 查看候选详情并审阅；
5. 把一个结果作为参考创建第二分支；
6. 保存、重新载入并核对模型、参考顺序、结果、来源和状态；
7. 从结果进入现有加工和导出路径。

**工程门**：全部定向和全量自动化、lint/format、Provider 契约、i18n、UI scaling、保存重开、凭据/日志守护、`git diff --check` 和固定状态截图通过。

**记录**：只保留本地提交与测试日志；不写正式阶段报告，不更新 `CURRENT-STATE.md`，不使用 Computer Use、不打真实 API、不构建候选包、不请求项目所有者测试。自动化全绿后直接进入 Beta 0.5。

**内部状态**：`Beta 0.4 工程门通过；继续 Beta 0.5`。不单独向项目所有者交接。
