## M1.1-1 auto_k 量化策略扩展：kmeans（quantizer.gd）

**目标**：`PFQuantizer` 新增 `auto_k_strategy: "median_cut" | "kmeans"` 参数，默认 `median_cut`（保持 M1 行为零变化），`kmeans` 作为 k 值较大（>16）场景的质量增强可选项。

**技术实现指导**：
- k-means 实现：以 median cut 的输出作为初始聚类中心（避免随机初始化的不确定性，同时保证结果不劣于 median cut 基线）；迭代上限 16 次或中心位移 < 0.5/255 提前收敛。
- 距离度量复用 palette.gd 已有的 OKLab 距离函数，不要重写——在 OKLab 空间聚类，质量优于 RGB 空间且代码已就位。
- 性能保护：采样而非全像素聚类。像素数 > 65536 时均匀降采样到 ≤ 65536 个样本点再聚类（聚类中心确定后仍对全图映射）。
- `PFCleanupParams` 契约更新：新增 `auto_k_strategy` 字段，默认值 `"median_cut"`；`normalize_params()` 注释同步说明语义（"质量 vs 速度的用户选择，kmeans 仅在 quantize=auto_k 时生效"）。
- UI：检查器量化区在 `auto_k` 模式下出现策略下拉框，默认 median_cut，tooltip 说明"k 较大时 kmeans 颜色分布更均匀但更慢"。
- **确定性要求**：相同输入 + 相同参数必须产出逐像素一致的输出（初始中心来自 median cut 已保证；禁止引入随机种子）。

**验收标准**：
1. 单测：同一渐变图，k=32 时 kmeans 输出的量化误差（OKLab 均方误差）≤ median cut 输出（断言 kmeans 不劣于基线）。
2. 单测：`auto_k_strategy` 缺省或非法值时回退 median_cut，输出与 M1 既有黄金测试逐像素一致（零回归硬断言）。
3. 确定性：同输入同参数连续运行 3 次输出逐像素一致。
4. 性能：512×512 图 k=32 kmeans 全流程 < 1.5s（自动化环境放宽 2 倍）。

## M1.1-2 自定义调色板 UI 路径（文件选择器 + 注册）

**目标**：让非技术用户无需写代码即可导入并使用自定义调色板。core 层 `PFPaletteRegistry.resolve()` 已支持外部 JSON 与颜色数组，本任务只做 UI 接线，不改 core 解析逻辑。

**技术实现指导**：
- 检查器调色板下拉框末尾增加"导入自定义调色板…"项，触发 `FileDialog`（过滤 `*.json`，后续格式如 Lospec `.hex`、`.gpl` 列为 future，meta 中记录不做）。
- 导入流程：选择文件 → `PFPaletteRegistry.resolve()` 解析 → 解析失败弹出明确错误信息（哪一行/哪个字段非法，不要静默失败）→ 成功后注册进 registry 并写入项目目录（`palettes/` 子目录，路径进 manifest——同步 `PROJECT-FORMAT.md`）。
- 导入的调色板随项目持久化：重新打开项目后下拉框中仍可见、可选。这是验收硬项——只在内存注册不算完成。
- 下拉框中自定义调色板与内置调色板视觉区分（分组或前缀），并提供删除入口（仅删自定义项，内置项不可删）。
- 色板预览：下拉项 hover 或选中时显示颜色条缩略预览（复用现有调色板渲染逻辑，若 M1 没有则实现一个 ≤ 32 色横条的最小版本）。

**验收标准**：
1. 集成测试：合法 JSON 导入 → registry 可 resolve → fixed_palette 清洗使用该调色板输出色数与色值正确。
2. 集成测试：非法 JSON（缺字段 / 色值越界 / 非数组）导入 → 报错信息含具体原因，registry 状态无污染。
3. 持久化测试：导入 → 保存项目 → 重新加载 → 调色板仍在且可用于清洗。
4. 人工评审：非技术用户视角走通"下载一个 Lospec JSON → 导入 → 清洗一张图"全流程无需文档辅助。

## M1.1-3 core 覆盖率可验证统计（工具链验证 + 诚实替代）

**目标**：把 M1 完成报告中"代码行数比"替换为可验证的覆盖率数字，对齐 `QUALITY.md` 的 `core/` 行覆盖 ≥80% 要求；若工具链不可行，落地经认可的替代统计并写入质量文档。

**技术实现指导**：
- 第一步（timebox 半天）：验证 GUT coverage（GUT 9.6 有限版本）在 Godot 4.6 headless 下是否可用。可用 → 接入 `verify_m1_1.sh`，输出 `core/pixel` 行覆盖数字并断言 ≥80%。
- 不可用 → **不要伪造行覆盖**，改为"public API / 分支矩阵覆盖表"：枚举 `core/pixel` 每个公开方法与每个分支条件（参数枚举值 × 边界条件），逐项标注对应测试用例名，矩阵存为 `05-quality/COVERAGE-MATRIX-M1.md`；未覆盖项必须补测试或注明豁免理由。
- 无论哪条路径，结论（数字或矩阵 + 方法说明）回填 M1 完成报告与 `QUALITY.md`，明确标注统计方法——这是对 M1 前置项第 2 条"解释替代统计方法"承诺的兑现。
- 工具链调研结果（可用/不可用、版本、坑点）记录进 `04-research/RESEARCH-NOTES.md` 附录，M2 起复用同一口径。

**验收标准**：
1. 二选一达成：GUT coverage 数字 ≥80% 进入出口脚本断言；或覆盖矩阵无未豁免空洞且经人工评审签字。
2. 替代路径下：矩阵中标注的每个测试用例名真实存在且通过（脚本抽查比对，防矩阵与测试脱节）。
3. `QUALITY.md` 更新覆盖率统计口径条目，M2 之后里程碑直接引用。

## M1.1-4 顺延加固项收尾（base_size 先验流入 + 批量帧时间断言）

**目标**：收尾 M1 评估报告中两项未进验收补丁的顺延项，关闭 M1 评估清单。

**技术实现指导**：
- **StylePreset base_size → 检测器先验自动流入**（评估 #13）：`cleanup_inspector.gd` 构建 params 时读取当前项目 StylePreset 的 `base_size`，注入 `grid_detector` 的先验参数（已有 `_resolve_search_range` 支持，±30% 搜索窗口）。UI 显示"使用预设先验 16px"的小字提示，用户手动改 scale 后先验失效（手动优先）。**先检查 M1 验收补丁 B 是否已顺手实现，已实现则本项仅补断言。**
- **批量帧时间自动化断言**（评估 #4）：`tests/integration/` 新增批量压测用例——50 张 fixtures 图批量 Apply，断言主线程峰值帧时间 < 100ms（即 UI 不冻结的量化口径）且总耗时 < 60s（自动化环境放宽 2 倍）。帧时间采集用 `Performance.get_monitor(TIME_PROCESS)` 逐帧记录取峰值。

**验收标准**：
1. 单测：含 base_size 的 StylePreset 项目 → 检测器收到的搜索范围 = 先验 ±30%；无 base_size 时回退全范围（M1 行为不变）。
2. 集成测试：批量压测用例进入 `verify_m1_1.sh`，峰值帧时间与总耗时断言生效。
3. M1 完成度评估报告 17 项逐条标注状态（补丁A/B/C 完成 / M1.1 完成 / 明确移交 M2），无悬空项。
