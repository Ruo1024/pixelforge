# QUALITY.md — 测试策略、完成定义、风险登记

## 1. 完成定义（DoD）——任务卡与集成阶段通用底线

1. 代码合并前：gdlint/gdformat 零告警；新增公共函数有类型标注，关键职责/契约/非显然逻辑按需注释，不要求机械头注释。
2. 测试：卡内可自动化验收转为定向测试并通过；Beta 集成边界运行全量回归。标注“真实窗口/人工”的项目不得用 headless 结果替代；当前 Beta 0.6 在所有工程切片完成后才统一交给项目所有者人工测试。
3. 文档：行为影响契约的改动已按修订流程更新 `02-contracts/`；CHANGELOG 一行摘要。
4. 无新增 lint 豁免、无 `# TODO` 无主任务（TODO 必须带 backlog 条目编号）。
5. 性能预算相关卡：实测数字写入交付说明。

### 1.1 本地 agent 验证口径

如果仓库明确选择“不启用 GitHub Actions”，则本地 agent 运行统一验证脚本作为出口门控。脚本必须在项目根记录清楚，并至少覆盖：

1. lint / format
2. 全量自动化测试
3. headless 启动或导出模板检查

PixelForge M0 当前采用 `pixel/scripts/verify_m0.sh`。后续若恢复 GitHub Actions，应把本段口径改回 CI 绿灯，并保留本地脚本作为开发前自检。

### 1.2 Beta 0.3–0.5 历史验证节奏

Beta 0.3–0.5 三版原型冲刺采用“自动化开发、最终统一人工验收”：

1. 卡内只运行与同一整体切片直接相关的定向自动化；
2. 每个 Beta 出口运行一次全量回归和脚本截图；
3. 开发期间完全禁止使用 Computer Use，禁止逐卡/逐版本 agent 交互式冒烟；
4. Beta 0.3、Beta 0.4 不构建候选；Beta 0.5 收口后构建首次统一候选，人工失败后的修复允许递增候选号重建；
5. 真实付费云端、真机输入手感、视觉质量和跨显示器由项目所有者在三版完成后一次性验收。

脚本截图只能证明场景成功渲染以及没有明显空白、重叠或裁切，不能标记为人工通过。具体节奏以 `03-milestones/BETA-0.3-0.5-ROADMAP.md` 为准。

### 1.3 Beta 0.6 当前验证节奏

Beta 0.6 采用“整体切片自动化、最终唯一候选、项目所有者统一人工验收”：

1. B6-0 建立测试骨架；B6-1、B6-2 至 B6-4、B6-5 至 B6-7 是三个集成边界，每个边界必须清零定向测试并运行全量回归；
2. 开发期间完全禁止 Computer Use、真实付费 API 和逐卡/逐版本 agent 交互式体验测试；
3. English/简中 × `1080×560 / 1280×720 / 1440×900` × UI scale `1.0 / 1.25 / 1.5` 共 18 组几何自动化是工程硬门；
4. 七张固定截图必须从真实 ProjectService/UI action 和程序生成的许可安全素材构造，输出到 ignored `scratch/`，并由 manifest 自验；
5. 结果卡 All 筛选下 50 张全部存在、可见、可命中；无分页、无内部滚动、无尾项截断；
6. LOD 只改变显示详略；卡片/端口/连线世界几何不变；项目加载 50% 时倍率 UI 同帧显示 50%；
7. 只有 `verify_beta_0_6.sh` 全绿后才构建一次 `0.6.0-beta.1` macOS 候选；人工失败后的修复可递增候选号；
8. 真实云端、输入手感、最终视觉、内外屏和跨屏移动全部留给项目所有者一次性执行 `manual-test-beta0.6.md`。

具体尺寸、场景、截图名、防伪与阻断条件以 `03-milestones/BETA-0.6-PLAN.md §6–§8` 为唯一当前口径。脚本截图和几何断言仍不能替代人工视觉签收。

### 1.4 DoD 核查表模板

Beta 0.3/0.4 只保留历史提交和自动化日志，Beta 0.5 已为三版填写统一工程报告；当前 Beta 0.6 只在 B6-8 填写一次下表。状态只能写：`通过`、`不适用`、`延期登记`、`阻塞`。

状态语义必须分层使用：

以下三项是报告的**结论层级**，不是上表状态列的新枚举；表格单项仍只填写 `通过 / 不适用 / 延期登记 / 阻塞`。

- **工程通过**：代码、自动化与文档达到卡片要求；可以合流到本地集成分支，但项目所有者人工 `go` 前不得合并 `main`，也不代表真人体验或版本可发布。
- **人工通过**：任务卡标注的真实窗口/视觉/工作流已由验收人执行并签收。
- **发布通过**：当前版本出口门中的工程、人工与目标平台项全部通过。
- `延期登记` 只允许关闭非出口项；凡被当前里程碑或版本列为出口门的项目，`延期登记` 等同于**发布未通过**，不得继续称为对外 alpha/MVP/v1.0。

## 2. 测试金字塔

| 层 | 工具 | 范围 | 目标 |
|---|---|---|---|
| 单元 | GUT | core/ 全部算法与模型 | 行覆盖 ≥80%，黄金用例为主 |
| 集成 | GUT | services/ + 契约（项目格式 round-trip、provider 契约、插件装卸）| 关键路径全覆盖 |
| UI 自动化 | GUT 场景/集成测试 | 组件实例化、状态、数据映射、关键交互序列 | 按风险覆盖，不以模块数量凑数 |
| 脚本 UI 证据 | GUT 场景 + 自动截图 | 固定数据、语言、窗口尺寸、状态和画布倍率 | 每个 Beta 集成边界 1 次；不代表体验通过 |
| 项目所有者人工 | docs/manual-test-*.md | 真实云端、真机体验、视觉质量、产品 go/no-go | Beta 0.6 全部工程完成后统一执行 |

**黄金测试方法论**（算法卡的统一模式）：fixtures 由 `tests/fixtures/generators/*.gd` 程序生成（已知真值），禁止手工 PNG 当真值（不可维护）；真实 AI 图样本仅用于人工评审项。

**契约测试**：PROVIDER-API/PLUGIN-API 各有参数化契约套件，新实现挂进来即跑全套——保证"接口即法律"可执行。

### 2.1 M1.1 core 覆盖率替代口径

Godot 4.6.3 + 当前 vendored GUT 版本未暴露可直接接入 headless 出口脚本的行覆盖率输出。M1.1 不伪造行覆盖数字，改用 `05-quality/COVERAGE-MATRIX-M1.md` 作为可验证替代统计：逐项列出 `core/pixel` 公开 API、关键分支、边界条件和对应 GUT 测试名；出口脚本必须抽查矩阵中的测试名真实存在，并禁止保留未覆盖空洞。M2 之后若升级测试工具链，可在不降低矩阵口径的前提下补回行覆盖数字。

## 3. 自动化流水线

当前 PixelForge M0 使用本地 agent 门控；下列 CI 流水线是未来恢复 GitHub Actions 时的目标形态。

```
push → lint → unit+integration (headless, Linux) → 冒烟 (headless) →
导出检查 (Win/macOS/Linux) → [标签构建] 三平台产物 + 冒烟启动
```
- 主分支保护：未来启用 GitHub Actions 后，CI 绿 + 一次评审仍只是工程条件；当前 Beta 0.6 还必须取得项目所有者人工 `go` 才可合并 `main`。未启用 CI 时，本地验证与评审只允许合流到本地集成分支。
- 性能哨兵：M0-3/M1-5/M5-1 的性能断言用宽松上限（自动化机器波动 2 倍裕量），本地严格值人工确认。
